# ADR 0004: Three key types — write, read, personal

**Status:** accepted

## Context

Different callers need very different blast radii. SDK snippets ship inside
apps and web pages, so their credential must be safe to expose. CI scripts
and internal dashboards need to *read* analytics without a human logging in.
Automation sometimes needs the full management API. One almighty "API key"
would force the most-exposed credential to carry the most privilege.

## Decision

Three prefixed key types, generated by `ApiKeyGenerator` (16 random bytes,
hex):

| Key | Prefix | Scope | Storage |
| --- | --- | --- | --- |
| Project write key | `pk_live_` | `POST /capture` and `POST /decide` only | plaintext on the project row (lookup key) |
| Project read key | `rk_live_` | Query endpoints + flag local-evaluation, via `X-Api-Key` | plaintext on the project row |
| Personal key | `pk_user_` | Everything the owning user can do, via `Authorization: Bearer` | **SHA-256 hash only**; plaintext shown once at creation, suffix kept for display |

The prefix does double duty: it makes key types visually unmistakable in
configs and logs, and it routes authentication — a policy scheme dispatches
`Bearer pk_user_…` to the personal-key handler and everything else to JWT
validation, so both credential styles share one header.

## Consequences

- A leaked write key can pollute data and read flag decisions but cannot read
  a single event. A leaked read key can read analytics for one project but
  cannot write or manage anything. The high-privilege personal key is the
  only one stored hashed and is revocable per key.
- Key type confusion fails closed and is tested: a write key on a query
  endpoint is 401, a read key on capture is 401, a read key on management
  endpoints is 401, keys never cross projects.
- Personal keys inherit membership dynamically — they see exactly the
  projects their owner sees, including the 404-not-403 behavior, and revoking
  one takes effect on the next request.
- Project keys being stored plaintext is a deliberate trade-off: they are
  lookup keys (the row is *found by* the key), lower-privilege, and shown in
  the project response; rotating them would be a schema-trivial follow-up.
